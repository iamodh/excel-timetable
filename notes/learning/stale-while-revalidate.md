# stale-while-revalidate — 캐시 갱신과 사용자 대기시간

`use-cache-and-tags.md`가 `revalidateTag(tag, profile)`의 두 번째 인자(`"max"` 등)를 어떻게 쓰는지 정리했다면, 이 문서는 그 인자가 의미하는 **SWR 캐시 사이클**을 개념으로 정리한다.

---

## 1. SWR이 풀려는 문제

캐시가 무효화된 **직후 첫 요청**을 어떻게 처리할 것인가?

- 옵션 A: 새 fetch가 끝날 때까지 사용자를 기다리게 한다 → 그 한 명은 로딩 화면을 본다.
- 옵션 B: 일단 **이전(stale) 값을 즉시 응답**하고, 백그라운드에서 새 fetch를 돌려 캐시를 교체한다.

옵션 B가 stale-while-revalidate. "사용자는 절대 기다리지 않는다, 대신 무효화 직후 한 번은 약간 구 데이터를 본다"는 트레이드.

---

## 2. 캐시의 3가지 상태

| 상태 | 응답에 사용 가능? | 응답 후 동작 |
|------|------------------|--------------|
| **fresh** | ✅ | 아무 일도 안 함 |
| **stale** | ✅ (즉시 반환) | 백그라운드에서 새 fetch → 캐시 교체 |
| **expired** | ❌ | 동기 fetch (사용자 대기) |

핵심: stale은 "여전히 응답 가능한" 상태다. expired는 폐기됨. 둘은 다르다.

---

## 3. 두 가지 타이머 + 외부 트리거

캐시가 fresh → stale로 넘어가는 경로는 세 가지:

1. **`revalidate` 시간 경과** — 자연 stale 전환. cacheLife 프로파일이 정한다.
2. **`revalidateTag(tag, ...)` 호출** — 시간 무관하게 즉시 stale 표시. 관리자 버튼·Server Action 등.
3. **`expire` 시간 도달** — fresh든 stale이든 무조건 폐기. 다음 요청은 동기 fetch.

`revalidate`와 `expire`는 cacheLife 프로파일이 한 쌍으로 정의한다 (예: `15m / 1y`).

---

## 4. 사이클 시각화

```
[빌드/첫 fetch]
      │
      ▼
[fresh 캐시]
      │
      ├──(revalidate 시간 경과)─────┐
      └──(revalidateTag 호출)───────┤
                                    ▼
                               [stale 표시]
                                    │
                              (다음 요청 1번)
                                    │
                       ┌────────────┴────────────┐
                       │                         │
                       ▼                         ▼
              응답: stale 즉시 반환       백그라운드 fetch
                                                 │
                                                 ▼
                                         캐시 교체 → [fresh]
                                         (또 사이클 반복)

[expire 시간 도달] ──> 폐기 ──> 다음 요청은 동기 fetch
```

중요한 두 가지:
- stale 응답은 **무효화 직후 첫 요청 1회만** 일어난다. 그 요청이 트리거한 백그라운드 fetch가 끝나면 캐시는 다시 fresh.
- `expire`는 "백그라운드 갱신이 실패하거나 요청이 한 번도 안 들어와서 stale이 너무 오래 방치"되는 케이스의 안전망.

---

## 5. cacheLife 프로파일 — 두 타이머의 길이 묶음

Next.js 16의 `revalidateTag(tag, profile)` 두 번째 인자.

빌트인 (짧은 → 긴 순서):
- `"seconds"`, `"minutes"`, `"hours"`, `"days"`, `"weeks"`, `"max"`
- 커스텀: `{ stale: ..., revalidate: ..., expire: ... }` (초 단위)

`"max"` = `revalidate`와 `expire` 둘 다 가장 길게. 자연 stale 전환이 거의 안 일어나고, 안전망 expire도 매우 멀리(예: 1년) 둔다.

---

## 6. 어떤 프로파일을 쓸까

판단 기준은 단순하다:

| 케이스 | 권장 프로파일 | 이유 |
|--------|---------------|------|
| 명시적 트리거가 진실원 (관리자 버튼, Server Action) | `"max"` | 시간 안전망이 발화할 일이 없음. 길수록 사용자 로딩 짧음 |
| 외부에서 자주 바뀌는데 트리거가 없음 | `"minutes"` / `"hours"` | 자연 stale 전환이 데이터 신선도를 책임 |
| 라이브 데이터 (주가, 점수) | `"seconds"` 또는 커스텀 | 신선도 우선, 캐시는 부하 분산 목적 |
| 거의 안 바뀜 (정적 콘텐츠) | `"days"` / `"weeks"` | 백그라운드 fetch 횟수 절감 |

직관: **트리거가 있으면 길게, 없으면 짧게.**

---

## 7. 이 프로젝트가 `"max"`를 고른 이유

시간표·공지 둘 다 진실원이 명시적이다:
- 시간표: 매니저가 시트 수정 → 관리자 페이지 "최신화" 버튼 → `revalidateTag("timetable", "max")`
- 공지: 관리자가 작성/삭제 → `revalidateTag("notice", "max")`

자연 stale 전환에 기댈 필요가 없으므로 시간 안전망은 가장 길게 두고, 학생 입장에서 **stale 즉시 응답이 가장 빠르게 일어나도록** SWR 창을 최대로 잡았다. expire(기본 1년)는 진짜 마지막 안전망 — "관리자가 6개월 동안 한 번도 안 누르는" 비현실적 시나리오용.

---

## 8. 자주 헷갈리는 포인트

### "`"max"`는 stale 값을 영원히 유지하는 거 아닌가"
아니다. stale 응답은 **무효화 직후 첫 요청 1회만**이다. 그 요청이 백그라운드 fetch를 트리거하고, 끝나면 캐시는 새 값으로 fresh가 된다. `"max"`가 길게 잡는 건 **다음 자연 stale 전환까지의 시간**이지, "stale 상태로 머무는 시간"이 아니다.

### "SWR이면 사용자는 항상 구 데이터를 본다"
아니다. 구 데이터를 보는 건 **무효화 직후 첫 요청자 한 명뿐**이다. 그 요청이 백그라운드 갱신을 일으켜 두 번째 요청자부터는 새 데이터를 본다.

### "프로파일이 길수록 서버 비용이 든다"
오히려 반대. 프로파일이 짧으면 자연 stale 전환이 자주 일어나 백그라운드 fetch가 자주 트리거된다. 길수록 fetch 횟수가 줄어든다. 단 외부 데이터 신선도가 떨어지는 트레이드.

### "expire와 revalidate 차이가 헷갈린다"
- `revalidate`: "이 시간 지나면 다음 요청 응답에 stale 마킹하고 백그라운드 갱신" — 사용자 응답 지연 0
- `expire`: "이 시간 지나면 캐시 폐기, 다음 요청은 동기 fetch" — 사용자 응답 지연 발생
  
expire는 거의 안 발화하는 안전망이어야 한다. 자주 발화하면 SWR의 "절대 기다리지 않는다" 약속이 깨진다.

### "`revalidateTag`를 호출하면 즉시 새 데이터로 응답되지 않나"
아니다. 호출 시점에 일어나는 건 **stale 마킹**뿐. 실제 새 fetch는 다음 요청이 들어와야 백그라운드에서 시작된다. 관리자가 버튼을 누른 직후에는 캐시에 변화가 없는 것처럼 보일 수 있다 — 다음 학생 요청이 트리거.
