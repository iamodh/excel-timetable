# 쿠키 (Cookies) 정리

M11(PIN 접근 제어)에서 학생 인증 상태를 쿠키로 유지하면서 정리한 노트.

---

## 1. 쿠키의 본질

**브라우저가 스스로 보관하고, 매 요청마다 자동으로 서버에 붙여 보내는 key-value 저장소.**

핵심은 "자동으로"다. 서버는 딱 한 번 `Set-Cookie` 헤더로 "이거 저장해둬"라고 시키기만 하면, 이후부터 브라우저가 알아서 매 요청 헤더에 `Cookie: key=value`를 붙인다. 개발자가 매번 JS로 붙일 필요가 없다.

이 "자동 포함" 특성 때문에 로그인 세션 유지, PIN 인증 같은 용도에 전통적으로 쓰인다.

---

## 2. 동작 흐름

```
① 학생이 /pin 페이지에서 "1234" 입력 후 submit
   ┌──────────────────────────────────────────────────┐
   │ POST /api/auth/pin                               │
   │ Content-Type: application/json                   │
   │                                                  │
   │ {"pin":"1234"}                                   │
   └──────────────────────────────────────────────────┘
                        │
                        ▼
   [서버] Upstash에서 저장된 PIN 조회 → 일치 확인
                        │
                        ▼
② 서버 응답 + Set-Cookie 헤더
   ┌──────────────────────────────────────────────────┐
   │ HTTP/1.1 200 OK                                  │
   │ Set-Cookie: student_pin=1234; HttpOnly; Path=/;  │
   │             SameSite=Lax; Max-Age=2592000        │
   │ Content-Type: application/json                   │
   │                                                  │
   │ {"message":"인증되었습니다."}                      │
   └──────────────────────────────────────────────────┘
                        │
                        ▼
   [브라우저] 로컬 쿠키 저장소에 기록
             - 도메인: example.com
             - 경로: /
             - 만료: 지금 + 30일

③ 이후 같은 도메인 요청마다 자동으로 붙음
   ┌──────────────────────────────────────────────────┐
   │ GET /                                            │
   │ Cookie: student_pin=1234           ← 자동 포함    │
   └──────────────────────────────────────────────────┘
                        │
                        ▼
   [middleware.ts] 쿠키 추출 → Upstash 저장값과 비교
                   일치: 통과 / 불일치: /pin 리다이렉트
```

## 3. Set-Cookie 속성 해부

이 프로젝트가 내려주는 쿠키:

```
Set-Cookie: student_pin=1234; HttpOnly; Path=/; SameSite=Lax; Max-Age=2592000
```

| 속성 | 값 | 의미 | 왜 이렇게? |
|------|------|------|-----------|
| 이름=값 | `student_pin=1234` | 브라우저가 저장할 key-value | middleware가 이 이름으로 읽음 |
| `HttpOnly` | — | JS(`document.cookie`)로 읽기 불가 | XSS 공격으로 PIN 탈취 방지. 서버만 읽음 |
| `Path=/` | — | 사이트 전체 경로에 쿠키 동봉 | `/`, `/pin`, `/api/...` 어디로 가든 쿠키가 따라감 |
| `SameSite=Lax` | — | 외부 사이트→이 사이트 요청 시 쿠키 제한 | CSRF 방어. 카카오톡에서 링크 타고 들어올 때 호환성 최고 |
| `Max-Age=2592000` | 30일 (60×60×24×30) | 만료까지 남은 초 | TECHSPEC §1.3 "쿠키 만료 30일" |

### SameSite 값의 차이

| 값 | 외부→내부 요청 시 | 쓰는 경우 |
|-----|------------------|----------|
| `Strict` | 절대 쿠키 안 보냄 | 은행, 결제 등 최고 보안 |
| **`Lax`** | **GET 요청(링크 클릭, 주소창 입력)엔 보냄. POST엔 안 보냄** | **일반 웹 앱 (이 프로젝트)** |
| `None` | 항상 보냄 (단, `Secure` 필수) | 크로스 사이트 iframe, 광고 |

카카오톡 인앱 브라우저에서 공유 링크를 누르면 GET 요청이 되므로 `Lax`로 충분하다.

### 안 쓴 속성들

| 속성 | 왜 안 썼나 |
|------|----------|
| `Secure` | HTTPS에서만 보내도록 제한. Vercel은 기본 HTTPS라 결국 같은 효과 + 로컬 개발(HTTP)에서 쿠키가 안 뜨는 문제 발생. 배포 단계에서 재검토 |
| `Domain=` | 생략 시 "쿠키 발급한 도메인으로 자동 고정" → 서브도메인 확장할 일 없으면 생략이 안전 |
| `Expires=` | `Max-Age`와 역할 중복. 최신 브라우저에선 `Max-Age`가 우선이라 생략 |

## 4. 브라우저 저장 방식

쿠키는 브라우저 내부 **도메인별 전용 저장소**에 보관된다. Chrome 기준:

- Mac: `~/Library/Application Support/Google/Chrome/Default/Cookies` (SQLite DB)
- 개발자 도구 → **Application 탭 → Storage → Cookies** 에서 시각적으로 확인

### HttpOnly의 가시성

`HttpOnly` 플래그가 있어도:
- ❌ `document.cookie`에서 안 보임 (JS 코드로 접근 불가)
- ✅ 개발자 도구 Application 탭에선 보임 (디버깅 용도)
- ✅ 브라우저가 Request 헤더에 자동으로 붙여 보냄

**즉 XSS 방어용**이지 "사용자에게 숨기는 것"이 아니다.

---

## 5. 다른 브라우저 저장소와의 비교

| | 쿠키 | localStorage | sessionStorage |
|---|------|--------------|----------------|
| 크기 한도 | 4KB | ~5MB | ~5MB |
| 서버로 자동 전송 | ✅ 매 요청마다 자동 | ❌ 수동 (fetch 등) | ❌ 수동 |
| JS 접근 | HttpOnly면 불가 | 항상 가능 | 항상 가능 |
| 만료 제어 | Max-Age/Expires | 수동 삭제만 | 탭 닫으면 소멸 |
| 용도 | **인증/세션** | 클라이언트 상태, 캐시 | 탭 내 임시 데이터 |

**PIN 인증은 쿠키가 맞다** — 왜냐하면:
1. middleware(서버)에서 매 요청마다 읽어야 함 → 자동 전송이 필수
2. JS에서 접근할 필요 전혀 없음 → HttpOnly로 XSS 방어 가능
3. 30일 만료 필요 → Max-Age로 깔끔히 관리

localStorage에 PIN을 넣으면 XSS 한 방에 탈취되고, middleware에서 읽을 수도 없다.

## 6. 이 프로젝트의 보안 설계

### 핵심: 쿠키 값 vs Redis 저장값의 분리

- **쿠키**: 학생이 가진 "입장권"
- **Upstash Redis**: "현재 유효한 PIN의 진실원 (source of truth)"

middleware는 **매 요청마다** `쿠키값 === Redis값`을 비교한다. 덕분에:

> **관리자가 M13에서 PIN을 변경하면, 기존 학생들의 세션이 즉시 전부 무효화된다.**

- Redis: `1234` → `5678`로 변경
- 학생 브라우저 쿠키: `student_pin=1234` (그대로 남음)
- middleware: `1234 !== 5678` → `/pin`으로 리다이렉트 → 재입력 요구

만약 "쿠키에 단순 플래그(`authed=true`)만 두고 서버에선 저장 PIN을 안 보는" 순진한 설계였다면, PIN이 유출되어 바꿔도 기존 사용자들은 계속 접근할 수 있었을 것이다. TECHSPEC §1.3이 "PIN은 KV 스토어에 저장 (관리자 페이지에서 변경 가능)"이라고 명시한 이유가 여기 있다.

### 현재 구현의 트레이드오프

쿠키에 **PIN 평문**(`student_pin=1234`)을 담는다. XSS는 HttpOnly로 막히지만, 개발자 도구/프록시로는 여전히 보인다.

| 대안 | 장점 | 단점 |
|------|------|------|
| **평문 PIN (현재)** | 단순. middleware 로직도 단순 | 네트워크 도청 시 노출 (HTTPS면 실제론 보호) |
| 해시 저장 | 쿠키에 평문 없음 | middleware 로직 복잡 |
| 세션 토큰 발급 | 쿠키에 PIN 흔적 전혀 없음, revoke 가능 | 세션 저장소 필요 (복잡도 증가) |

**현재 맥락(공유 PIN, 관리자 1~2명, 신뢰 관계, HTTPS 전제)에선 평문 + HttpOnly로 충분**하다는 판단. TECHSPEC도 해싱/세션 토큰을 요구하지 않는다. 다만 이 트레이드오프를 기억해둘 것.

## 7. 자주 헷갈리는 포인트

### "쿠키는 브라우저에 저장되니까 클라이언트 기술이다?"

반은 맞고 반은 틀리다. **저장 위치는 브라우저지만, 주도권은 서버에 있다.** 서버가 `Set-Cookie`로 내려보내고, 만료 시각과 속성을 정한다. 브라우저는 그 지시대로 보관하고 되돌려보낼 뿐이다.

### "HttpOnly 쿠키는 서버만 볼 수 있다?"

정확히는 **"JS 코드는 볼 수 없지만, 브라우저 자신은 알고 있다"**. 브라우저가 요청 헤더에 자동으로 붙여 보낼 수 있는 이유다. 개발자 도구로도 보인다.

### "Cookie vs Set-Cookie"

| 방향 | 헤더 이름 | 쓰는 쪽 |
|------|----------|---------|
| 서버 → 브라우저 | **`Set-Cookie`** | 서버가 응답에 포함. "이거 저장해둬" |
| 브라우저 → 서버 | **`Cookie`** | 브라우저가 요청에 자동 포함. "너가 저장하라고 했던 거" |

### "쿠키를 여러 개 내려보내려면?"

`Set-Cookie` 헤더를 **여러 번** 응답에 포함하면 된다. 세미콜론으로 이어붙이면 안 된다:

```
Set-Cookie: student_pin=1234; HttpOnly; Path=/
Set-Cookie: theme=dark; Path=/
```
