## WORKFLOW

### `go` - 구현 사이클

1. **Check**: `git status`로 미커밋 변경사항 확인 — 있으면 먼저 커밋 또는 사용자에게 보고
2. **Test First**: 가장 첫 번째 미완료 테스트 케이스를 작성한다
3. **Minimal Code**: 테스트를 통과시키는 최소한의 코드만 작성한다
4. **Lint**: `npm run lint` 전체 실행 (ESLint + TypeScript 타입 체크)
5. **Run Tests**: `npm test` 실행
6. **Report & Wait**: 결과를 보고하고 사용자 확인을 기다린다

> ⚠️ 이후 자동 진행 금지. 사용자 피드백에 따라 수정하거나 `commit`을 기다린다.

### `commit` - 완료 처리

1. **코드 커밋**: 변경사항을 성격별로 나누어 커밋한다 (리팩토링, 컴포넌트 분리, 기능 등)
2. **PLAN 체크**: 완료된 항목에 체크박스 표시 + `- Commits:` 줄에 커밋 해시 추가 (매 항목마다 커밋하지 않고, 마일스톤 완료 시 또는 사용자 요청 시 한 번에 커밋)

### `verify` - 마일스톤 완료 후 수동 검증

마일스톤 완료 시 사용자가 직접 확인할 수 있는 검증 명령어를 제안한다.

**제안 범위:**
- 브라우저에서 확인할 수 있는 URL과 기대 동작 (UI 마일스톤인 경우)
- `npm run lint`, `npm test` 등 CLI 명령어
- Chrome DevTools 모바일 에뮬레이터 확인 (모바일 관련 마일스톤)

**규칙:**
- `commit` 직전 또는 직후에 제안
- 해당 마일스톤에서 구현한 기능에 한정 (이전 마일스톤 내용 반복 금지)
- 복사-붙여넣기로 바로 실행 가능한 형태로 제공

### 명령어

| 명령어     | 동작                                 |
| ---------- | ------------------------------------ |
| `go`       | 구현 사이클 실행 후 사용자 확인 대기 |
| `commit`   | 커밋 및 PLAN.md 체크박스 표시        |
| `status`   | 현재 마일스톤 진행 테이블 + 요약 + 다음 항목 출력 |
| `verify`   | 마일스톤 완료 후 수동 검증 제안      |
| `refactor` | 테스트 통과 후 리팩토링 제안         |

### 필수 실행 명령어

```bash

# 테스트 실행 (통과할 때까지 반복)
```

## TESTING

> 코드 예제는 **TECHSPEC.md § 8** 참조

### 원칙

1. **Next.js를 믿어라**: 프레임워크가 보장하는 건 테스트하지 마 (라우팅, RSC 렌더링 등)
2. **비즈니스 규칙만 테스트해라**: 시트 파싱, 색상 변환, 병합 처리, rate limit
3. **목 데이터 활용**: Google Sheets API 응답을 목 데이터로 만들어 파싱 로직 테스트
4. **동일 성격 검증은 하나의 테스트로 합쳐라**: 필수 필드 3개 → 테스트 1개에서 모두 검증 (필드별 분리 금지)
5. **테스트 내 인라인 로직을 대체하는 함수가 추가되면, 기존 테스트도 해당 함수를 사용하도록 수정한다**
6. **기대값은 고정값을 사용한다**: 테스트 안에서 쿼리로 기대값을 만들지 말 것 — 테스트는 고정적인 문서 역할을 한다
7. **mock-in mock-out 테스트 금지**: mock 데이터를 넣고 같은 mock 데이터가 나오는지 확인하는 테스트는 작성하지 않는다 — 실제 로직 검증이 없는 테스트는 무의미하다

### 테스트 작성 원칙

- 유닛 테스트는 `lib/` 내 순수 함수에 집중 (파싱, 변환, 유틸)
- API Route Handler 테스트는 요청/응답 플로우 검증에만 사용
- 컴포넌트 렌더링은 수동 테스트로 대체 (Vitest만 사용, React Testing Library 미도입)

## CONSTRAINTS

### npm 사용 규칙

- 새로운 패키지를 사용하기 전에 반드시 `package.json`에 존재하는지 확인
- 없으면 코드 작성 전에 추가 필요 여부를 먼저 알려줄 것
- 기존 의존성만으로 해결 가능한지 우선 검토

- 테스트 없이 프로덕션 코드 작성 금지
- 한 번에 여러 기능 구현 금지
- TECHSPEC.md에 명시되지 않은 기술 스택 도입 금지
- 테스트 통과 전 리팩토링 금지
- 미리 추상화 금지 — 중복이 실제로 발생한 후에 추출한다
- 기능 추가 시 UI에서 접근 가능하도록 버튼/링크를 반드시 함께 제공한다 — 브라우저에서 verify할 수 있어야 한다

### 기존 기능 대체 시

- 새 함수/모듈을 구현했다면 같은 작업 단위 내에서 호출 지점을 교체해 실제로 사용한다 — 구현만 하고 기존 경로가 계속 쓰이는 상태로 남기지 않는다
- 기존 테스트가 이미 검증하는 동작은 새 구조에서 재검증하지 않는다 — 기존 테스트를 새 구조에 맞춰 갱신하면 충분하다

### BRANCH RULES

- main 브랜치에서 직접 작업한다 (별도 feature 브랜치 생성 금지)
- 에이전트는 커밋을 수행하되, push는 하지 않는다

## STACK

> 기술 스택 상세는 **TECHSPEC.md § 4.1** 참조

### Coding Conventions

- **들여쓰기:** 2 spaces
- **네이밍:** camelCase (변수/함수), PascalCase (컴포넌트/타입)
- **커밋 메시지:** Conventional Commits 형식
- **커밋 분리:** 문서 수정(CLAUDE.md, PLAN.md 등)과 기능 추가(코드, 테스트)는 별도 커밋으로 분리한다
- **Server/Client 분리:** 서버 컴포넌트가 기본, `"use client"`는 상태/이벤트가 필요한 컴포넌트에만 사용한다
- **데이터/비즈니스 규칙은 `lib/` 레이어가 보호한다:** 파싱, 변환, rate limit 등의 규칙은 반드시 해당 레이어에서 검증한다
